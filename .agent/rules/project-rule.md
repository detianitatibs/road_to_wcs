# プロジェクトのフォルダ構成(概要)
大まかなプロジェクトのフォルダ構成は以下の通り従ってください。

## docs フォルダ
トップディレクトリ配下に存在します。
日本語のMarkdownファイルでユーザ要件定義や必要な設計書を管理するフォルダです。

## apps フォルダ
トップディレクトリ配下に存在します。
各ツールのアプリケーションを作成するときは、このフォルダの配下にアプリケーション単位でフォルダを作成します。

## .github フォルダ
トップディレクトリ配下に存在します。
Github ActionsのCICDパイプラインを管理するフォルダです。

# プロジェクトの進め方

プロジェクトは以下のサイクルで進めることを想定しています。

1. 要件整理フェーズ
  1. まずはやりたいことの壁打ちを実施し、明確化します
  2. ドメイン駆動設計にしたがって、要望をドメインモデルに落とし込んで壁打ちをします。
  3. 決定事項はdocsフォルダの **ユーザー要件定義設計書** に反映します。
2. 要件定義フェーズ
  1. 要件をもとにユーザーストーリーを作成します
  2. ユーザーストーリーの内容が決まると、docsフォルダの **ユーザーストーリー一覧** に反映します。
3. 開発フェーズ
  1. ユーザーストーリー単位で開発を実施します
  2. 必ずユーザーストーリー単位でbranchを作成します
  3. 品質保証のためにテストを実行し、成功したことを確認した後にプルリクエストを作成し、管理者にレビューを依頼します

これらの1-3のフェーズを何度も繰り返します。

# 開発方針

開発する際は以下の方針を採用します。

- AI-SDLCに沿った開発手法を採用します
- ブランチ戦略は **Github Flow** を採用します
